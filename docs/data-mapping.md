# WooCommerce → Shopify data mapping

The migration scripts in `shopify/migration_from_woocommerce/` implement this mapping. Source is WooCommerce (WordPress) at `lush.qa`; target is the Shopify dev store `lush-qatar.myshopify.com`. The migration is done **in-house** (no third-party migration tools). Scope baseline: up to 500 products / 5,000 customers / 5,000 orders — flag if actual volumes exceed this.

The site is bilingual (Arabic/English), mirroring the Lush KSA store; capture translations for every translatable entity.

## Source-verified facts (live API review, 2026-07-19)

Findings from inspecting real `wc/v3` payloads that shape the mapping below:

- **Multilingual:** WPML. `lang=en|ar` works on the REST API; records carry a `translations` map linking sibling IDs. Arabic slugs are percent-encoded — decode before use.
- **SEO:** no SEO plugin fields (no Yoast/RankMath) in payloads. Theme is XStore (`_et_*` meta) + Elementor. SEO title/description must be generated by fallback rule (product name + stripped description excerpt).
- **Brands:** Woo `brands` taxonomy in use; sampled value "Lush" → Shopify vendor.
- **Category hierarchy:** 44 of 61 categories have a parent. Shopify collections are flat — see Categories section for the flattening rule.
- **Field usage (sample of 100 products):** `short_description` 0, SKU 86, `weight` 9, GTIN (`global_unique_id`) 0. **14% of products have no SKU** — order line items must link by `product_id`/`variation_id` first, SKU only as fallback.
- **Reading wp-admin counts:** the WP products list shows combined-language totals (WPML): "All (1,065) ≈ EN 538 + AR 526 (+1 stray), Published (653), Drafts (412), Trash (12)" — roughly double the per-language REST counts. The REST API with an explicit `lang` gives the real per-language numbers; the tool's verify stage records per-language counts, and final go-live numbers come from its verify snapshot plus Shopify's own counts, not from wp-admin.
- **Product statuses (EN, via REST):** 319 published, **219 draft (40% of the catalog)**, 4 trashed. Drafts migrate as Shopify DRAFT products (hidden from the storefront, but orders still link to their variants — this is exactly the "previously sold, later set to draft" case). Trashed products are not migrated; order lines referencing them become custom line items.
- **Guest orders are common** (2 of 3 recent orders had `customer_id: 0`).
- **PDF invoice plugin** (`wpo_wcpdf_invoice_number`) maintains a separate client-facing invoice number sequence — preserve it as a metafield.
- Customer `meta_data` contains legacy `opencart_pass`/`opencart_salt` (the Woo store was itself migrated from OpenCart once) — ignore.

## Customers

| WooCommerce | Shopify | Notes |
|---|---|---|
| Email | Email | Dedup key (lowercased); skip records without email |
| First / last name | First / last name | |
| Company | Address company | |
| Billing address (addr 1 & 2, city, postcode, country, state) | Default address | |

| `role`, `username`, `avatar_url`, `is_paying_customer` | — | Ignore |

> Passwords do not migrate. Decide whether to trigger Shopify account-invite/reset flow at cutover.
> Marketing consent is never set by migration (no implicit opt-ins).
> Guest purchasers (orders with `customer_id: 0` — common on lush.qa) are not created as customers in v1; their orders carry the billing email only.

## URL redirects (SEO preservation)

- Map **all** existing WooCommerce URLs (products, categories, pages, blog) to their Shopify equivalents.
- Create **301 redirects** in Shopify for every changed path.
- Preserve handles wherever possible to minimize redirects.
